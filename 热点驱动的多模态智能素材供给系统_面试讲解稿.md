# 热点驱动的多模态智能素材供给系统：面试讲解稿与高频问题

## 一、完整面试讲解稿

这个项目我主要是围绕两个业务需求来设计的。

第一个需求是用户自己的素材库检索。用户会上传很多自己的素材，比如视频、图片、音频、文档等，但是当用户真正想找素材的时候，他的描述往往不是准确关键词，而是自然语言描述。比如用户可能会说：“帮我找一个适合夏天饮品广告的清爽视频”，或者“找一个有科技感、城市夜景、节奏比较快的开场素材”。这种搜索如果只靠传统关键词检索，很容易搜不到，因为用户描述的是场景、风格、情绪和使用目的，而素材标题或标签里不一定包含这些词。

所以我对第一个需求的理解是，它本质上不是普通的素材管理系统，而是一个多模态语义检索系统。用户上传素材以后，系统需要对素材做自动理解。比如视频要用 FFmpeg 抽帧、切片、提取音频；音频要用 Whisper 做语音转文字；图片和视频关键帧要做 OCR、caption 和视觉 embedding；标题、字幕、OCR、人工标签、评论等文本内容要进入全文索引和文本向量库。最后原始素材放 MinIO，结构化元数据放 MySQL，标题、字幕、OCR、评论等内容放 Elasticsearch，文本、图片、视频片段的 embedding 放到向量数据库，比如 Qdrant、Milvus 或 Weaviate。

查询的时候，用户输入的自然语言不会直接拿去查库，而是先通过 LLM 做 Query Rewrite，把模糊描述拆成素材类型、场景、风格、情绪、时长、版权状态、是否要求热点等结构化条件。然后系统同时做关键词召回、向量召回和结构化过滤。比如 Elasticsearch 负责精确关键词、字幕、OCR、标签、时间、平台、版权过滤；向量数据库负责语义相似度匹配；最后再结合热度、质量、版权、用户偏好做重排序。

这里我会特别注意视频素材的检索粒度。视频不能只按整个文件生成一个向量，因为用户经常要找的是某几秒的镜头，比如“年轻人举杯欢笑”或者“城市夜景快闪”。所以我会把视频拆成片段级 AssetSegment，每个片段保存开始时间、结束时间、关键帧、字幕、画面描述和 embedding。这样用户搜索的时候，系统可以直接返回某个视频的具体时间段，而不是只返回整个视频。

第二个需求是热点追踪和外部素材采集。这个需求的目标不是等用户上传素材，而是系统主动追踪互联网上最近正在爆的内容，比如趋势榜、热搜榜、新闻 RSS、短视频热点、评论区舆情等，然后判断这个热点为什么火、和内部业务有没有关系、是否值得沉淀成候选素材。

我会把这部分设计成一个热点内容雷达系统。系统通过定时任务触发多个数据源适配器，比如 Google Trends、新闻 RSS、行业网站、短视频热点榜、视频平台 API、社交媒体公开榜单等。采集策略上我会分层选择，优先使用官方 API 或 RSS，因为这类数据稳定、结构化、合规性更好；如果没有 API，就先用 web-fetch 或 httpx 读取静态 HTML；如果静态页面没有有效正文，再升级到 Chrome DevTools MCP 或 Playwright 加载动态页面，通过 take_snapshot 获取渲染后的页面文本结构。

但是热点内容采集不能只停留在网页正文。真正有价值的爆点素材，往往存在于封面图、视频片段、评论区、标题、页面截图、互动数据和网络请求里面。所以在动态页面加载之后，我还会结合 DOM 解析、evaluate_script、network request 监听和页面截图，提取图片、视频、封面、音频、评论、点赞数、转发数、播放量、发布时间等多模态信息。

在多模态提取层，我会按照素材类型分别做技术选型。网页正文用 web-fetch、httpx、BeautifulSoup、lxml、readability 或 trafilatura 提取；动态页面用 Chrome DevTools MCP 或 Playwright 兜底；图片用 PaddleOCR 提取图中文字，用 CLIP、SigLIP 或 BGE-VL 生成视觉向量；视频用 FFmpeg 抽帧、切片、提取音频，再用 Whisper 做语音转写；视频画面则通过关键帧 embedding 或视频 embedding 模型做片段级向量化；评论和互动数据进入 MySQL 或 Elasticsearch，用于热点评分、舆情分析和排序。

外部采集到的内容不会直接进入用户私有素材库。因为这里涉及版权风险、质量问题、重复内容和用户素材库污染。所以我会设计三层素材池：第一层是热点候选库，外部采集内容先进入 candidate 状态；第二层是审核后的公共素材库，通过版权、质量、风险、重复度校验之后才能进入；第三层是用户私有素材库，只有用户上传、用户收藏、用户购买授权或用户确认使用的素材，才进入用户自己的向量库。

这样两个业务逻辑就能打通成一个闭环。热点系统先发现外部爆点，生成素材需求；系统先去用户自己的素材库里检索，如果内部素材足够，就直接推荐给用户；如果内部素材不够，再从外部采集候选素材，经过多模态理解、版权过滤、去重和质量评估之后，再进入候选素材池或公共素材池。后续用户再搜索的时候，这些素材也可以通过自然语言被检索出来。

技术架构上，我会让 Go 或 Java 负责主业务系统，比如用户、权限、素材管理、任务状态、检索接口和内部推送；Python 负责采集、网页解析、OCR、Whisper、多模态模型和 embedding 服务，因为 Python 在爬虫和 AI 模型生态上更成熟。中间通过 Kafka 解耦采集、解析、向量化、审核和入库流程。存储层用 MinIO 保存原始文件、截图、封面和关键帧；MySQL 保存素材元数据、版权状态、任务状态和热度分；Elasticsearch 保存标题、正文、字幕、OCR、评论等全文内容；向量数据库保存文本、图片、视频片段的 embedding；Redis 用来缓存热点榜单、搜索结果和任务状态。

所以这个系统的核心价值不是简单地“爬热点”，也不是简单地“搜素材”，而是把热点发现、内容采集、多模态理解、版权过滤、向量化入库和自然语言检索串成一个完整的智能素材供给链路。对业务来说，它可以降低运营、剪辑和内容团队找素材的成本，提高热点响应速度，并且把外部爆点内容转化成内部可检索、可复用、可管理的素材资产。

## 二、面试 1 分钟精简版

这个项目我主要解决两个问题：第一个是用户自己的多模态素材库怎么通过自然语言检索，第二个是系统怎么主动追踪互联网热点，并把热点内容沉淀成可复用的候选素材。

对于用户素材库，我不会只做关键词搜索，而是会对视频、图片、音频、文本做多模态理解。比如视频用 FFmpeg 抽帧和切片，用 Whisper 做语音转写，用 OCR 提取画面文字，用 CLIP、BGE-VL 或视频 embedding 模型生成向量。原始文件放 MinIO，元数据放 MySQL，标题、字幕、OCR、评论放 Elasticsearch，embedding 放 Qdrant 或 Milvus。用户搜索时，先用 LLM 做 Query Rewrite，把自然语言拆成场景、风格、情绪、素材类型、版权状态等条件，再做关键词召回、向量召回、结构化过滤和重排序。

对于热点采集，我会通过 API、RSS、静态网页、动态网页分层接入。优先用官方 API 或 RSS，没有的话先用 web-fetch 抓静态 HTML，如果正文不足，再升级到 Chrome DevTools MCP 或 Playwright 加载动态 DOM。除了正文，我还会提取封面图、视频、评论、互动数据、页面截图和网络请求里的媒体资源，再经过 OCR、Whisper、关键帧抽取、caption、embedding、舆情分析和热度评分，进入候选素材池。

最后这两个业务会形成闭环：热点系统发现爆点后，先查用户私有素材库；如果素材不足，再从外部采集候选素材；通过版权、质量、风险和去重校验后，再进入公共素材库或用户素材库。这个系统的价值是把热点发现、素材采集、多模态理解、向量化检索和素材复用打通，降低内容团队找素材的成本，提高热点响应速度。

## 三、面试官高频问题与回答要点

### 1. 你这个项目解决的核心业务问题是什么？

核心解决两个问题。

第一，用户素材库越来越大之后，传统关键词检索很难找到合适素材。因为用户描述的是场景、情绪、风格和用途，比如“清爽”“科技感”“年轻化”，不一定是素材标题里的关键词。

第二，热点内容变化很快，人工追热点和找素材效率低。所以系统需要主动发现热点、采集候选素材、分析爆点原因，并把外部内容转化成内部可检索的素材资产。

一句话总结：

> 这个项目不是简单素材管理，而是热点驱动的多模态智能素材供给系统。

---

### 2. 为什么传统关键词搜索不够？

因为关键词搜索只能匹配显式文本字段。

比如用户搜索：

> “适合夏天饮品广告的清爽视频。”

素材标题可能叫：

> “户外聚会 003.mov”

标题里没有“夏天”“饮品”“清爽”，但画面里可能有阳光、冰饮、年轻人、浅色背景。这种场景需要图像理解、视频理解和语义向量检索。

关键词搜索适合精确匹配，向量检索适合语义匹配，所以我会做 Hybrid Search。

---

### 3. 为什么要做多模态检索？

因为素材本身不是纯文本。

视频里有画面、字幕、音频、封面、评论、互动数据；图片里有视觉风格、文字、人物、场景；音频里有语音和情绪。如果只存标题和标签，会丢失大量信息。

所以需要把文本、图片、视频片段、音频转写都转成可检索的语义表示。

---

### 4. 用户上传一个视频后，系统怎么处理？

我会走异步处理链路：

```text
用户上传视频
→ MinIO 保存原文件
→ MySQL 记录素材元数据
→ Kafka 投递解析任务
→ FFmpeg 抽帧、切片、提取音频
→ Whisper 做语音转写
→ OCR 提取画面文字
→ 图片 / 关键帧生成 caption 和视觉向量
→ 字幕 / OCR / 标题进入 Elasticsearch
→ 文本 / 图片 / 视频片段 embedding 进入向量数据库
→ 更新素材状态为可检索
```

这样可以避免上传接口被模型处理阻塞。

---

### 5. 为什么视频要做片段级检索？

因为用户找的往往不是整个视频，而是视频里的某个镜头。

比如一个 60 秒视频，只有 5 秒是“城市夜景快闪”，如果只给整个视频生成一个 embedding，这个特征会被稀释。片段级检索可以直接返回：

```text
video_001
12.5s - 17.5s
匹配原因：城市夜景、蓝色灯光、快节奏剪辑
```

这样对运营和剪辑人员更有价值。

---

### 6. 片段怎么切？

可以有三种方式：

第一种是固定时间窗口，比如每 3 秒或 5 秒切一段。

第二种是基于镜头边界切分，比如画面变化明显时切分。

第三种是结合字幕或语义变化切分，比如一句话或一个完整动作作为一个片段。

早期可以先用固定窗口 + 关键帧抽取，后续再优化成镜头检测和语义切分。

---

### 7. 为什么需要 LLM Query Rewrite？

因为用户输入很口语化，不适合直接查库。

比如用户说：

> “找最近比较火的，适合年轻人情绪营销的城市夜景视频。”

LLM 可以解析成：

```json
{
  "asset_type": "video",
  "scene": ["城市夜景"],
  "emotion": ["孤独感", "情绪营销"],
  "audience": ["年轻人"],
  "trend_required": true,
  "time_range": "recent",
  "copyright_status": "usable"
}
```

这样后面才能做向量召回、关键词召回和结构化过滤。

---

### 8. 为什么要用 Hybrid Search？

因为关键词检索和向量检索解决的问题不一样。

Elasticsearch 适合精确匹配，比如品牌名、人名、标签、字幕、OCR、平台、发布时间。

向量检索适合语义匹配，比如“清爽感”“高级感”“年轻化”“科技风”。

Hybrid Search 可以同时提高召回率和准确率。

---

### 9. Elasticsearch 和向量数据库怎么分工？

Elasticsearch 负责：

```text
标题
正文
字幕
OCR
评论
标签
关键词搜索
时间过滤
平台过滤
版权过滤
```

向量数据库负责：

```text
文本 embedding
图片 embedding
视频片段 embedding
语义相似度检索
跨模态检索
```

一句话：

> ES 解决“文字和条件怎么查”，向量库解决“语义相似怎么找”。

---

### 10. 为什么不直接用 Elasticsearch 存向量？

可以用，但要看业务规模。

如果素材量不大，ES 的 kNN 能满足需求，可以减少系统复杂度。

但如果视频片段很多、多模态向量很多、需要更灵活的 payload filter、多向量字段和高性能 ANN 检索，我会选择 Qdrant、Milvus 或 Weaviate 这类专门向量数据库。

---

### 11. Qdrant、Milvus、Weaviate 怎么选？

我会这样考虑：

```text
Qdrant：部署简单，payload filter 好用，适合中小规模业务快速落地。
Milvus：适合大规模向量检索，性能和生态成熟。
Weaviate：对象模型比较完整，多模态模块和 schema 管理比较方便。
```

如果是项目早期，我倾向 Qdrant；如果数据量非常大，比如海量视频片段，可以考虑 Milvus。

---

### 12. 多模态 embedding 模型怎么选？

文本可以用：

```text
BGE-M3
bge-large-zh
text-embedding 系列
```

图片可以用：

```text
CLIP
SigLIP
BGE-VL
Visualized-BGE
```

视频可以分阶段：

```text
早期：关键帧 CLIP 聚合 + 字幕文本向量
进阶：InternVideo / VideoCLIP / ImageBind / TwelveLabs API
```

音频早期不一定单独做音频 embedding，先用 Whisper 转写成文本，进入文本检索链路。

---

### 13. 为什么视频不一开始就上复杂视频大模型？

因为成本和工程复杂度比较高。

早期可以用更稳定的方式：

```text
FFmpeg 抽关键帧
→ 关键帧用 CLIP / BGE-VL 做视觉向量
→ 音频用 Whisper 转写
→ 字幕文本做文本向量
→ 片段级聚合
```

这样已经能覆盖大部分检索场景。后续如果业务对动作、镜头、时序理解要求更高，再接 InternVideo、VideoCLIP、ImageBind 或商业视频理解 API。

---

### 14. 热点系统的核心流程是什么？

核心流程是：

```text
定时任务触发
→ 采集趋势榜 / 热搜榜 / RSS / API
→ 解析热点标题、链接、热度、时间
→ 抓取详情页
→ 提取正文、图片、视频、评论、互动数据
→ 去重
→ 聚类
→ 热度评分
→ 舆情分析
→ 多模态理解
→ 候选素材入库
→ 推送给内部系统
```

重点是：不仅要知道什么火，还要知道为什么火，以及能不能转化成素材。

---

### 15. 热点数据源怎么选？

我会按优先级选择：

```text
官方 API / RSS
> 静态网页抓取
> 动态浏览器渲染
> 第三方数据服务
```

原因是 API 和 RSS 更稳定、更结构化、合规风险更低。网页抓取容易受页面改版、登录、反爬影响，所以不应该作为第一选择。

---

### 16. 为什么先 web-fetch，再升级 Chrome DevTools MCP？

这是出于性能和成本考虑。

web-fetch 或 httpx 读取静态页面速度快、成本低，适合第一层尝试。如果返回正文为空、正文太短、关键字段缺失，说明页面可能依赖 JS 渲染，这时再升级到 Chrome DevTools MCP 或 Playwright。

这样可以避免所有页面都启动浏览器，降低资源消耗。

---

### 17. take_snapshot 能解决什么，不能解决什么？

take_snapshot 主要解决：

```text
页面渲染后有哪些文本
有哪些可访问元素
正文区域大概在哪里
按钮、标题、列表结构是什么
```

但它不能完整解决：

```text
视频真实地址
图片真实地址
m3u8 请求
接口 JSON
页面所有网络资源
复杂评论分页
```

所以还需要配合：

```text
evaluate_script
network request 监听
get_network_request
take_screenshot
DOM 解析
```

---

### 18. 多模态内容具体从网页里提取什么？

我会提取五类内容：

```text
文本：标题、正文、摘要、话题、评论、作者、发布时间
图片：封面图、正文图、海报图、截图、视频缩略图
视频：video/source、m3u8、mp4、封面、时长、关键帧
音频：视频音轨、语音内容、背景音
互动数据：点赞、评论、转发、收藏、播放量、增长速度
```

这些内容分别进入不同处理链路，最后统一成 Asset 模型。

---

### 19. 图片内容怎么处理？

图片提取后我不会只保存 URL，而是会做四件事：

```text
保存原图或截图到 MinIO
用 OCR 提取图片里的文字
用 VLM 或 caption 模型生成图片描述
用 CLIP / SigLIP / BGE-VL 生成视觉 embedding
```

这样用户可以用自然语言搜索图片，比如：

> “蓝色科技感海报”  
> “年轻人聚会场景”  
> “夏天清爽风格封面”

---

### 20. 视频内容怎么处理？

视频处理链路是：

```text
获取视频地址或文件
→ 保存封面和元数据
→ FFmpeg 抽帧、切片、提取音频
→ Whisper 做语音转写
→ OCR 提取关键帧文字
→ 关键帧生成视觉 embedding
→ 字幕生成文本 embedding
→ 片段级入库
```

如果外部平台不允许下载原视频，就只保存 URL、封面、截图、摘要、embedding 和来源信息，作为候选素材，而不是直接保存原视频。

---

### 21. 评论区为什么重要？

因为评论区能反映爆点原因和舆情倾向。

比如同一个视频，画面本身普通，但评论区可能大量讨论某个梗、某种情绪或某个争议点。这些信息可以帮助系统判断：

```text
为什么火
情绪是正向还是负向
是否有争议风险
适合什么内容方向
是否值得推荐给运营
```

所以评论可以进入 ES 和 LLM 分析链路，用于热点评分和摘要。

---

### 22. 外部采集到的素材能不能直接放进用户私有库？

不能。

外部素材有版权、质量、重复和安全风险。如果直接放进用户私有库，会污染用户素材库，也可能带来合规问题。

我会设计三层：

```text
candidate：热点候选库
public_usable：审核后的公共素材库
user_private：用户私有素材库
```

外部素材先进入候选库，通过审核或用户确认后，才能进入正式素材库。

---

### 23. 用户私有库和公共热点库怎么隔离？

我推荐“公共库 + 私有库”模式。

```text
user_private_vectors：用户私有素材
public_hot_vectors：公共热点候选素材
```

查询时：

```text
先搜用户私有库
再搜公共热点库
最后合并排序
```

这样既能保护用户数据，又能用热点素材补充素材供给。

---

### 24. 多租户向量库怎么设计？

有三种方案。

第一种，一个 collection，payload 里带 user_id，查询时 filter user_id。优点是简单，适合早期。

第二种，每个用户一个 collection。优点是隔离强，缺点是 collection 数量多，维护复杂。

第三种，公共库 + 用户私有库。公共热点素材放公共库，用户上传素材放私有库。这个方案更适合业务系统。

---

### 25. 怎么做内容去重？

我会分模态去重。

文本去重：

```text
URL 去重
标题归一化
SimHash
MinHash
文本 embedding 相似度
```

图片去重：

```text
感知哈希
图片 embedding 相似度
尺寸和文件指纹
```

视频去重：

```text
关键帧相似度
视频片段 embedding 相似度
音频指纹
来源 URL 去重
```

热点去重：

```text
关键词聚类
标题聚类
跨平台 topic 聚类
```

---

### 26. 怎么做热点评分？

热点评分不能只看绝对热度。

我会综合：

```text
当前热度
热度增长速度
发布时间
平台权重
互动量
评论情绪
争议程度
业务关键词相关性
素材可用性
版权状态
```

比如一个热点热度很高但已经过期，价值可能不如一个最近 2 小时快速上升的话题。

---

### 27. 怎么做热点聚类？

可以用三种方式：

第一，关键词规则聚类，比如同义词、品牌词、事件词。

第二，文本 embedding 聚类，把标题、正文、评论摘要转向量，再用 KMeans、HDBSCAN 或相似度阈值聚类。

第三，LLM 辅助聚类，让 LLM 判断多个热点是否属于同一事件。

工程上可以先用 embedding 聚类 + 规则兜底，LLM 用在难判断或最终摘要阶段。

---

### 28. 为什么要用 Kafka？

因为采集、解析、多模态处理和向量化都比较耗时，不适合同步执行。

Kafka 可以把链路拆开：

```text
采集任务 topic
解析任务 topic
OCR 任务 topic
ASR 任务 topic
Embedding 任务 topic
审核任务 topic
入库任务 topic
```

这样每个模块可以独立扩容，也能避免某个模型服务慢导致整个系统阻塞。

---

### 29. 失败重试怎么设计？

不同任务要有不同策略。

```text
网页请求失败：重试 + 限速 + 代理池或降级
动态页面失败：回退静态内容或截图
OCR 失败：标记部分失败，不影响整体入库
Whisper 失败：保留视频和封面，稍后补偿
Embedding 失败：进入待向量化队列
版权校验失败：只保留候选信息，不进入正式库
```

要避免一个子任务失败导致整个素材完全不可用。

---

### 30. 怎么评价检索效果好不好？

可以从三个层面评估。

第一是召回效果：

```text
Recall@K
Precision@K
MRR
nDCG
```

第二是业务效果：

```text
用户点击率
收藏率
下载率
素材采用率
搜索后人工修改次数
平均找素材时间
```

第三是结果质量：

```text
是否返回具体视频片段
匹配原因是否合理
版权状态是否准确
重复素材是否少
```

最终还是要看内容团队是否真的更快找到可用素材。

---

### 31. 怎么做重排序？

初召回之后可以综合打分：

```text
最终分数 =
语义相关性
+ 关键词匹配
+ 热度分
+ 视觉质量
+ 版权可用性
+ 用户偏好
+ 时间新鲜度
```

比如：

```text
语义相关但版权不可用 → 降权
热度高但和用户需求无关 → 降权
用户私有素材匹配度高 → 优先
视频片段比整条视频更精准 → 优先
```

---

### 32. 怎么处理版权问题？

我会把版权状态作为素材的核心字段。

```text
internal：用户自己上传
authorized：已授权
public_candidate：公开候选
unknown：未知
restricted：不可用
```

外部采集内容默认不能直接当成可用素材，只能先进入候选库。正式推荐给用户下载或使用之前，需要经过版权来源、授权状态和平台规则校验。

---

### 33. 怎么避免爬虫系统不稳定？

主要靠四点。

第一，数据源优先级设计：能用 API/RSS 就不用网页爬取。

第二，分层降级：静态 fetch 失败再动态渲染，动态失败保留截图和元数据。

第三，适配器模式：每个平台独立 HotSourceAdapter，页面改版只影响单个平台。

第四，监控和告警：采集成功率、解析成功率、字段缺失率、任务积压、接口耗时都要监控。

---

### 34. 为什么 Go / Java 做主业务，Python 做模型服务？

因为两边优势不一样。

Go / Java 更适合做：

```text
用户权限
素材管理
任务状态
接口服务
并发调度
内部系统集成
```

Python 更适合做：

```text
爬虫
网页解析
OCR
Whisper
FFmpeg 调度
多模态模型
embedding 服务
```

中间用 Kafka 或 HTTP/gRPC 解耦，这样技术栈更合理。

---

### 35. 这个项目最大的工程难点是什么？

我觉得有三个。

第一，多源数据接入难。不同平台结构不同，需要统一 HotItem 和 Asset 模型。

第二，多模态处理链路长。文本、图片、视频、音频的处理方式不同，需要异步解耦和状态管理。

第三，检索质量和合规控制难。既要搜得准，又不能把低质量、重复或有版权风险的素材直接推给用户。

## 五、最强收尾句

面试最后可以这样总结：

> 我对这个系统的理解是，它不是单纯的爬虫，也不是单纯的素材库，而是一个热点驱动的多模态智能素材供给系统。热点追踪解决“内容从哪里来”，多模态提取解决“内容里有什么”，向量化检索解决“用户怎么找到”，候选库和版权审核解决“素材能不能安全复用”。这几个环节打通以后，才能真正把外部爆点内容沉淀成内部可检索、可复用、可管理的素材资产。
