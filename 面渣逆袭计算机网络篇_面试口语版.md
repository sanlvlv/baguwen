# 面渣逆袭计算机网络篇：面试口语版

> 整理方式：每题都按“先给结论，再解释原因，最后补充细节”的方式写，方便面试时直接说出来。

---

## 基础

### 1. 说下计算机网络体系结构

计算机网络体系结构常见有三种说法：OSI 七层模型、TCP/IP 四层模型和五层模型。

OSI 七层模型更偏理论标准，从上到下分别是应用层、表示层、会话层、传输层、网络层、数据链路层和物理层。TCP/IP 四层模型更偏实际落地，从上到下是应用层、传输层、网际层和网络接口层。五层模型通常是学习时为了方便理解，把 TCP/IP 和 OSI 做了折中，分成应用层、传输层、网络层、数据链路层和物理层。

面试里我一般会这么说：OSI 是理论模型，TCP/IP 是实际互联网主要使用的模型，五层模型是教学和理解网络通信过程时常用的模型。

---

### 2. 说一下每一层对应的网络协议有哪些？

应用层常见协议有 HTTP、HTTPS、DNS、FTP、SMTP、POP3、IMAP、Telnet、SSH 等。它们主要服务具体应用，比如网页访问、域名解析、文件传输、邮件收发和远程登录。

传输层主要是 TCP 和 UDP。TCP 面向连接，可靠传输；UDP 无连接，传输开销小，速度快，但是不保证可靠性。

网络层主要是 IP、ICMP、ARP、RARP、IGMP 等。IP 负责寻址和路由，ICMP 常用于错误报告和 ping，ARP 用来根据 IP 找 MAC 地址。

数据链路层常见有 Ethernet、PPP 等协议，负责相邻节点之间的数据帧传输。物理层主要关注网线、光纤、电信号、比特流这些底层传输细节。

---

### 3. 数据在各层之间是怎么传输的？

数据传输可以理解成“发送方层层封装，接收方层层拆封”。

发送方从应用层开始，把用户数据交给传输层，传输层加上 TCP 或 UDP 头部；再交给网络层，网络层加上 IP 头部；再交给数据链路层，加上帧头和帧尾；最后物理层把它转换成比特流发出去。

接收方刚好反过来。物理层收到比特流，数据链路层去掉帧头帧尾，网络层去掉 IP 头部，传输层去掉 TCP 或 UDP 头部，最后应用层拿到真正的数据。

可以用寄快递类比：每经过一层，就套一层包装，并写上这一层需要的地址和控制信息；到了目的地之后，再一层一层拆开。

## 网络综合

### 4. 从浏览器地址栏输入 URL 到显示主页的过程？

这个过程可以分成几个核心步骤：DNS 解析、建立 TCP 连接、发送 HTTP 请求、服务器处理并返回响应、浏览器解析渲染页面，最后根据情况关闭连接。

具体来说，浏览器输入 URL 后，首先会通过 DNS 把域名解析成 IP 地址。拿到 IP 后，浏览器会和服务器建立 TCP 连接，HTTPS 场景下还会进行 TLS 握手。连接建立后，浏览器发送 HTTP 请求，服务器处理请求并返回 HTTP 响应。浏览器拿到 HTML、CSS、JS、图片等资源后，解析 DOM 树、CSSOM 树，生成渲染树，最后绘制页面。

面试时可以顺带展开：DNS 解析涉及缓存和多级域名服务器；TCP 建连涉及三次握手；HTTPS 涉及证书校验和密钥协商；页面加载还可能涉及重定向、缓存、CDN 和 TCP 四次挥手。

---

### 5. 说说 DNS 的解析过程？

DNS 的作用是把域名解析成 IP 地址。比如访问 `www.baidu.com`，浏览器并不会直接认识这个域名，它需要先查到对应的 IP。

解析过程一般是这样的：浏览器先查自己的 DNS 缓存，如果没有，再查操作系统缓存和 hosts 文件；还没有的话，请求会发给本地 DNS 服务器。本地 DNS 如果也没有缓存，就会依次请求根域名服务器、顶级域名服务器，比如 `.com`，然后再请求权威域名服务器，最后拿到 `www.baidu.com` 对应的 IP 地址。

拿到结果之后，本地 DNS 会缓存这个解析结果，浏览器也可能缓存。这样下次访问同一个域名时，就不用重新完整解析，提高访问速度。

---

### 6. 说说 WebSocket 与 Socket 的区别？

Socket 是网络编程接口，可以理解成 IP 地址、端口和协议的一种组合。它是对 TCP/IP 协议栈的一层封装，方便程序员进行网络通信开发。

WebSocket 是应用层协议，主要用来实现浏览器和服务器之间的全双工、持久化通信。它一般是先通过 HTTP 完成握手，然后升级为 WebSocket 连接，之后客户端和服务端都可以主动发送消息。

所以二者不是一个层面的东西。Socket 更偏底层网络编程接口，WebSocket 是应用层通信协议。面试时可以说：Socket 是“怎么编程通信”，WebSocket 是“浏览器和服务端如何保持长连接通信”。

---

### 7. 说一下你了解的端口及对应的服务？

常见端口需要记住几个高频的。21 是 FTP，22 是 SSH，23 是 Telnet，53 是 DNS，80 是 HTTP，443 是 HTTPS，3306 是 MySQL，6379 是 Redis，8080 经常用作 Web 服务的备用端口。

这些端口的作用是让主机上的不同服务能够被区分。IP 地址定位一台机器，端口定位这台机器上的某个具体服务。

面试里不用死背所有端口，但 HTTP 80、HTTPS 443、DNS 53、SSH 22、MySQL 3306、Redis 6379 这几个一定要熟。

## HTTP

### 8. 说说 HTTP 常用的状态码及其含义？

HTTP 状态码可以先按大类记。1xx 表示提示信息，2xx 表示成功，3xx 表示重定向，4xx 表示客户端错误，5xx 表示服务端错误。

常见的状态码有：200 表示请求成功；301 表示永久重定向；302 表示临时重定向；400 表示请求语法错误；401 表示未认证；403 表示服务器理解请求但拒绝执行；404 表示资源不存在；500 表示服务器内部错误；502 表示网关错误；503 表示服务暂时不可用。

301 和 302 的区别也很常问。301 是永久移动，以后应该访问新的地址；302 是临时移动，客户端下次仍然可以使用原地址。

---

### 9. HTTP 有哪些请求方式？

HTTP 常见请求方法有 GET、POST、PUT、DELETE、HEAD、OPTIONS、CONNECT、TRACE。

GET 通常用于获取资源，POST 通常用于提交数据，PUT 通常用于更新资源，DELETE 用于删除资源。HEAD 和 GET 类似，但只获取响应头，不返回响应体。OPTIONS 用来查询服务器支持哪些请求方法。

在 RESTful 风格里，GET、POST、PUT、DELETE 通常对应查、增、改、删。但实际项目里也要结合接口设计规范来看，不是绝对的。

---

### 10. 说一下 GET 和 POST 的区别？

GET 和 POST 可以从三个角度回答。

第一，从报文结构看，GET 通常把参数放在 URL 后面，POST 通常把数据放在请求体里。所以 GET 受到 URL 长度限制，POST 更适合传输较大的数据。从直观上看，GET 参数会暴露在 URL 中，POST 相对不那么直接暴露，但这不代表 POST 就天然安全，真正安全还要依赖 HTTPS。

第二，从语义上看，GET 用来获取资源，应该是安全和幂等的；POST 用来提交数据，可能会修改服务器资源，所以通常不是幂等的。

第三，从缓存角度看，GET 更容易被浏览器、代理服务器、CDN 缓存，也可以保存为书签；POST 一般不会被缓存，也不适合直接作为书签保存。

---

### 11. GET 的长度限制是多少？

HTTP 协议本身没有规定 GET 的固定长度限制。真正限制 GET 长度的通常是浏览器、服务器或者代理对 URL 长度的限制。

因为 GET 参数一般放在 URL 中，所以 URL 太长时可能会被浏览器或服务器拒绝。不同浏览器限制不同，比如早期 IE 支持的 URL 长度比较短，Chrome、Firefox 会更长一些。

所以面试时可以说：GET 长度限制不是 HTTP 协议本身限制，而是客户端、服务端和中间代理对 URL 长度的实际限制。

---

### 12. HTTP 请求的过程与原理？

HTTP 是应用层协议，它规定了浏览器如何向服务器请求资源，服务器如何把资源返回给浏览器。

基本过程是：客户端和服务器先建立 TCP 连接，然后客户端**发送 HTTP 请求报文**，服务器收到后处理请求，返回 HTTP 响应报文。客户端拿到响应后进行解析。如果是短连接，通信结束后会关闭 TCP 连接；如果是长连接，则连接可以继续复用。

HTTP 本身不负责可靠传输，底层依赖 TCP 来保证数据可靠到达。HTTP 主要定义的是请求和响应的格式、状态码、头部字段以及客户端和服务端交互规则。

---

### 13. 说一下 HTTP 的报文结构？

HTTP 报文分为请求报文和响应报文。

请求报文主要由请求行、请求头、空行和请求体组成。请求行包括请求方法、URL 和 HTTP 版本，比如 `GET /index.html HTTP/1.1`。请求头里会带 Host、User-Agent、Accept、Cookie 等信息。空行用于分隔头部和请求体，请求体常见于 POST 请求。

响应报文主要由状态行、响应头、空行和响应体组成。状态行包括 HTTP 版本、状态码和状态说明，比如 `HTTP/1.1 200 OK`。响应头里会有 Content-Type、Content-Length、Cache-Control、Set-Cookie 等字段。响应体就是服务器真正返回的内容，比如 HTML、JSON 或图片数据。

---

### 14. URI 和 URL 有什么区别？

URI 是统一资源标识符，用来唯一标识一个资源。URL 是统一资源定位符，是 URI 的一种子集，不仅能标识资源，还能告诉你怎么找到这个资源。

可以类比一下：URI 像身份证号，重点是唯一标识；URL 像家庭住址，除了能标识这个人，还能告诉你怎么找到他。

所以所有 URL 都是 URI，但不是所有 URI 都一定是 URL。

---

### 15. 说下 HTTP/1.0、HTTP/1.1、HTTP/2.0 的区别？

HTTP/1.0 默认是短连接，每次请求通常都要重新建立 TCP 连接，请求完成后连接就关闭，所以连接建立和关闭的成本比较高。当然 HTTP/1.0 也可以通过 `Connection: keep-alive` 开启长连接，但不是默认行为。

HTTP/1.1 默认支持长连接，一个 TCP 连接可以复用来发送多个 HTTP 请求，减少了频繁建立连接的开销。同时 HTTP/1.1 支持管道化，客户端可以连续发送多个请求；也支持分块传输编码，在不知道响应体总长度时可以边生成边传输。但是 HTTP/1.1 仍然存在队头阻塞，因为同一个连接上的响应通常要按顺序返回，前面的响应慢了，后面的响应也会被卡住。

HTTP/2.0 的核心改进是二进制分帧、多路复用、头部压缩和服务端推送。多路复用让多个请求和响应可以在一个 TCP 连接里并发交错传输，不需要像 HTTP/1.1 那样严格按顺序返回，所以提高了传输效率。头部压缩减少了重复请求头的开销。不过 HTTP/2 仍然基于 TCP，所以如果 TCP 层发生丢包，整个连接还是可能受到影响。

---

### 16. HTTP/3 了解吗？

HTTP/3 最大的变化是底层不再使用 TCP，而是使用基于 UDP 的 QUIC 协议。QUIC 在 UDP 之上实现了可靠传输、拥塞控制、TLS 加密和多路复用。

HTTP/2 虽然支持多路复用，但是它还是跑在 TCP 上。如果 TCP 层发生丢包，整个连接的数据交付都可能被阻塞，这就是 TCP 层的队头阻塞。HTTP/3 使用 QUIC 后，不同流之间相对独立，一个流丢包通常不会阻塞其他流，所以可以减少队头阻塞。

另外，QUIC 把连接建立和 TLS 加密握手整合在一起，可以减少握手次数，提高连接建立速度。它还支持连接迁移，比如移动端从 Wi-Fi 切换到 4G 时，从四元组源ip,源端口，目标ip到目标端口升级为连接id标识连接，连接更容易保持。

---

### 17. HTTP 如何实现长连接？在什么时候会超时？

HTTP 长连接本质上是 TCP 长连接的复用。也就是说，客户端和服务端建立一次 TCP 连接后，不是请求完成马上关闭，而是继续保持一段时间，后续多个 HTTP 请求都可以复用这条连接。

在 HTTP/1.0 中，长连接默认关闭，需要设置 `Connection: keep-alive`。从 HTTP/1.1 开始，长连接默认开启。

长连接不会无限存在。服务端一般会配置 keep-alive timeout，如果一段时间内连接没有新的请求，就会关闭。TCP 层也有 keepalive 机制，会在连接空闲一段时间后发送探测包，如果多次探测都没有响应，就认为连接失效并关闭。

---

### 18. 说说 HTTP 与 HTTPS 有哪些区别？

HTTP 是明文传输，数据在网络中可能被窃听、篡改或者伪造。HTTPS 是在 HTTP 和 TCP 之间加了一层 SSL/TLS，用来实现加密传输、身份认证和完整性校验。

连接过程也不同。HTTP 建立 TCP 连接后就可以直接传输报文；HTTPS 在 TCP 三次握手后，还需要进行 TLS 握手，完成证书验证和密钥协商后，才会开始加密通信。

端口也不同，HTTP 默认端口是 80，HTTPS 默认端口是 443。另外 HTTPS 需要服务器申请数字证书，用来证明服务器身份可信。

---

### 19. 为什么要用 HTTPS？解决了哪些问题？

HTTPS 主要是为了解决 HTTP 明文传输带来的安全问题。

HTTP 有三个典型风险。第一是窃听风险，别人可以在链路上看到通信内容；第二是篡改风险，别人可以修改传输内容，比如插入广告或恶意脚本；第三是冒充风险，用户可能访问到了伪造的网站。

HTTPS 通过 SSL/TLS 解决这些问题。加密机制保证内容不容易被窃听；完整性校验保证内容被篡改后能被发现；数字证书保证客户端访问的服务器是真实可信的。

---

### 20. HTTPS 工作流程是怎样的？

HTTPS 的核心流程是：先用非对称加密安全地协商出一个对称密钥，然后后续通信使用对称加密。

具体来说，客户端访问服务端的 443 端口，服务端会返回数字证书，证书里包含服务端公钥、证书颁发机构和有效期等信息。客户端收到证书后会校验证书是否合法。如果证书可信，客户端会生成一个随机对称密钥，并用服务端证书中的公钥加密后发给服务端。

服务端用自己的私钥解密，拿到这个对称密钥。之后客户端和服务端就使用这个对称密钥进行加密通信。这样做的原因是：非对称加密安全但慢，对称加密快，所以 HTTPS 把两者结合起来使用。

---

### 21. 客户端怎么去校验证书的合法性？

证书是由 CA 机构签发的。CA 会把服务端公钥、域名、有效期、颁发者等信息打包后做 Hash，然后用 CA 的私钥对这个 Hash 做签名，最终形成数字证书。

客户端校验证书时，会先用同样的 Hash 算法对证书内容计算出一个 Hash 值。然后浏览器或操作系统里内置了受信任 CA 的公钥，客户端会用 CA 公钥解开证书签名，得到另一个 Hash 值。

如果两个 Hash 值一致，说明证书内容没有被篡改，并且确实是被可信 CA 签发的。除此之外，还会检查证书是否过期、域名是否匹配、证书链是否可信等。

---

### 22. 如何理解 HTTP 协议是无状态的？

HTTP 无状态指的是服务端不会天然记住客户端上一次请求的状态。也就是说，同一个客户端连续请求两次，HTTP 协议本身不会告诉服务器“这是刚才那个用户”。

这样设计的好处是简单、可扩展，服务器不需要为每个连接保存复杂状态。但问题是实际业务里经常需要状态，比如登录状态、购物车、用户身份等。

所以 Web 应用通常用 Cookie、Session、Token 等机制来补充状态管理。HTTP 本身无状态，但应用层可以通过这些机制实现“有状态”的业务体验。

---

### 23. 说说 Session 和 Cookie 有什么联系和区别？

Cookie 是保存在客户端的一小段数据，浏览器会在访问同一域名时自动带上对应 Cookie。Session 是保存在服务端的会话数据，用来记录用户状态。

它们的主要区别是：Cookie 存在客户端，Session 存在服务端；Cookie 容量较小，一般单个 Cookie 几 KB；Session 可以存更多数据，但会占用服务端资源；Cookie 容易被客户端看到和修改，Session 相对更安全。

它们的联系是：Session 通常会借助 Cookie 来保存 SessionID。用户第一次访问时，服务端创建 Session，并把 SessionID 返回给浏览器。浏览器把 SessionID 存进 Cookie，下次请求时带上这个 SessionID，服务端再根据它找到对应 Session。

在分布式系统里，如果用户请求被负载均衡到不同服务器，Session 可能找不到。常见解决方案是把 Session 存到 Redis 这类分布式缓存中，多台服务器共享。

## TCP

### 24. 详细说一下 TCP 的三次握手机制

TCP 是面向连接的协议，正式传输数据前要先通过三次握手建立连接。

第一次握手，客户端发送 SYN 报文给服务端，表示想建立连接，客户端进入 SYN_SENT 状态。第二次握手，服务端收到后返回 SYN + ACK，表示自己收到了客户端请求，也同意建立连接，服务端进入 SYN_RCVD 状态。第三次握手，客户端收到服务端确认后，再发送 ACK 给服务端，之后客户端和服务端都进入 ESTABLISHED 状态，连接建立完成。

三次握手的核心是确认双方的发送能力和接收能力都正常。客户端知道自己能发、服务端能收；服务端知道自己能收、也能发；第三次握手后，服务端也知道客户端能收。

---

### 25. TCP 握手为什么是三次，为什么不能是两次？不能是四次？

三次握手是为了确认双方的收发能力，并避免无效连接占用服务器资源。

如果只有两次握手，服务端收到 SYN 后返回 SYN + ACK 就认为连接建立成功，但客户端可能没有收到服务端的回应。这样服务端会白白维护一个连接，浪费资源。另外，如果一个历史失效的 SYN 报文延迟到达服务端，两次握手也可能导致服务端误以为是新连接。

第三次握手的作用就是让客户端告诉服务端：“我已经收到你的确认了，我们可以真正建立连接。”

为什么不是四次？因为三次已经足够确认双方的发送和接收能力，再多一次不会带来必要收益，只会增加连接建立时间。

---

### 26. 三次握手中每一次没收到报文会发生什么情况？

如果第一次握手服务端没有收到客户端的 SYN，服务端不会有任何动作，客户端等待超时后会重传 SYN，直到超过最大重传次数后连接失败。

如果第二次握手客户端没有收到服务端的 SYN + ACK，客户端会继续重传 SYN。服务端这边已经进入 SYN_RCVD 状态，会等待客户端第三次握手的 ACK。

如果第三次握手服务端没有收到客户端 ACK，服务端会重传 SYN + ACK。如果一直收不到 ACK，最后会认为连接建立失败。客户端可能以为自己已经建立连接并开始发数据，但服务端没完成连接，此时服务端可能返回 RST 报文来重置连接。

---

### 27. 第二次握手传回了 ACK，为什么还要传回 SYN？

ACK 是服务端告诉客户端：“我已经收到了你发来的 SYN。”这是对客户端连接请求的确认。

SYN 是服务端告诉客户端：“我这边也想建立连接，并且我的初始序列号是这个。”

所以第二次握手里 SYN 和 ACK 含义不同。ACK 是确认客户端，SYN 是服务端发起自己的同步请求。两个合在一起发，可以减少一次通信。

---

### 28. 第 3 次握手可以携带数据吗？

第三次握手是可以携带数据的。因为对于客户端来说，收到服务端的 SYN + ACK 后，已经确认服务端的收发能力是正常的，客户端已经进入 ESTABLISHED 状态。

但是第一次握手不能携带数据。因为如果第一次 SYN 就允许携带大量数据，攻击者可以伪造大量 SYN 包并附带数据，让服务端消耗更多 CPU 和内存，容易造成攻击风险。

所以一般认为：第一次、第二次握手不应该携带应用数据，第三次握手理论上可以携带数据。

---

### 29. 说说半连接队列和 SYN Flood 攻击的关系？

服务端在监听状态下，会维护两个队列：半连接队列和全连接队列。

客户端发来 SYN 后，服务端返回 SYN + ACK，此时三次握手还没完成，这个连接会进入半连接队列，也叫 SYN 队列。等客户端再回 ACK，三次握手完成后，连接会进入全连接队列，也叫 accept 队列，等待应用程序 accept。

SYN Flood 攻击就是攻击者伪造大量 IP，向服务器发送大量 SYN 报文。服务端回复 SYN + ACK 后，迟迟收不到第三次握手的 ACK，于是半连接队列被大量占满，正常用户的连接请求就可能进不来。

常见防护方式有 SYN Cookie、调大半连接队列、缩短重传时间、使用防火墙或负载均衡清洗流量。

---

### 30. 说说 TCP 四次挥手的过程？

TCP 四次挥手用于断开连接。假设客户端主动关闭连接。

第一次挥手，客户端发送 FIN 报文，表示自己不再发送数据了，进入 FIN_WAIT_1。第二次挥手，服务端收到后返回 ACK，表示确认收到客户端的关闭请求，服务端进入 CLOSE_WAIT，客户端进入 FIN_WAIT_2。

第三次挥手，服务端把剩余数据处理完后，也发送 FIN 报文，表示服务端也要关闭连接，进入 LAST_ACK。第四次挥手，客户端收到服务端 FIN 后返回 ACK，并进入 TIME_WAIT。服务端收到 ACK 后关闭连接，客户端等待 2MSL 后进入 CLOSED。

---

### 31. TCP 挥手为什么需要四次呢？

因为 TCP 是全双工通信，双方的发送通道和接收通道需要分别关闭。

客户端发送 FIN，只表示客户端不再发送数据了，但它仍然可以接收服务端数据。服务端收到 FIN 后先回 ACK，表示知道客户端要关闭了。但是服务端可能还有数据没发完，所以不能马上发送 FIN。

等服务端的数据也发送完了，再单独发送 FIN 给客户端。这样服务端的 ACK 和 FIN 通常不能合并，所以关闭连接一般需要四次挥手。

---

### 32. TCP 四次挥手过程中，为什么需要等待 2MSL 才进入 CLOSED？

TIME_WAIT 等待 2MSL 主要有两个目的。

第一，保证客户端最后发送的 ACK 能到达服务端。如果这个 ACK 丢了，服务端会重传 FIN，客户端在 TIME_WAIT 状态还能收到并重新发送 ACK。如果客户端马上关闭，就**无法处理服务端重传的 FIN**。

第二，防止旧连接中的报文影响新连接。等待 2MSL 可以让网络中这个连接产生的旧报文基本都消失，避免下一个相同四元组的连接收到历史残留数据。

所以 TIME_WAIT 不是浪费，而是为了保证连接可靠关闭和避免旧报文干扰。

---

### 33. 保活计时器有什么用？

TCP 保活计时器是为了检测长时间空闲的连接是否还有效。

比如客户端和服务端建立了 TCP 连接，但客户端突然宕机或者网络断开，没有正常发送 FIN。服务端如果完全不检测，就可能一直维护这个无效连接，浪费资源。

启用 keepalive 后，连接空闲一段时间，服务端会发送探测报文。如果连续多次探测都没有收到响应，服务端就认为连接已经不可用，然后关闭连接。

---

### 34. CLOSE_WAIT 和 TIME_WAIT 的状态和意义？

CLOSE_WAIT 通常出现在被动关闭连接的一方。比如客户端发 FIN 给服务端，服务端收到后回 ACK，此时服务端进入 CLOSE_WAIT。这个状态表示服务端已经知道对方要关闭了，但自己可能还有数据要处理或发送。

TIME_WAIT 通常出现在主动关闭连接的一方。比如客户端最后发送 ACK 后进入 TIME_WAIT，并等待 2MSL 再关闭。它的意义是保证最后 ACK 能被服务端收到，同时避免旧报文影响后续新连接。

如果服务端出现大量 CLOSE_WAIT，通常说明程序没有正确关闭 socket。如果出现大量 TIME_WAIT，通常说明本机主动关闭了大量短连接。

---

### 35. TIME_WAIT 状态过多会导致什么问题？怎么解决？

TIME_WAIT 过多主要会占用内存和端口资源。如果短时间内大量主动关闭连接，本地端口可能被大量 TIME_WAIT 占用，导致新连接创建受影响。

解决思路有几个。第一，尽量使用长连接，减少频繁建立和断开 TCP 连接。第二，可以合理调整系统内核参数，比如端口范围、TIME_WAIT 复用相关配置。第三，服务端程序可以使用 `SO_REUSEADDR`，允许某些情况下复用地址和端口。

不过要注意，不能为了减少 TIME_WAIT 就随意缩短等待时间，因为 TIME_WAIT 本身是 TCP 可靠关闭机制的一部分。

---

### 36. 说说 TCP 报文首部的格式？

TCP 首部里比较重要的字段有源端口、目的端口、序列号、确认号、首部长度、标志位、窗口大小、校验和、紧急指针和选项字段。

源端口和目的端口用于标识通信双方的应用进程。序列号用于标记发送字节流的位置，保证数据有序。确认号用于告诉对方自己期望收到的下一个字节序号。

标志位也很重要，比如 SYN 用于建立连接，ACK 表示确认号有效，FIN 用于关闭连接，RST 用于重置连接，PSH 表示尽快交给应用层，URG 表示紧急指针有效。

窗口大小用于流量控制，告诉对方自己还能接收多少数据。校验和用于检测数据在传输过程中是否出错。

---

### 37. TCP 是如何保证可靠性的？

TCP 的可靠性不是靠单一机制，而是一整套机制共同保证的。

首先是连接管理，通过三次握手建立连接，通过四次挥手释放连接。其次是序列号和确认应答，发送方给每个字节编号，接收方通过 ACK 告诉发送方收到了哪些数据。

然后是校验和，用来检测报文在传输过程中是否出错。还有超时重传和快速重传，如果数据丢失，发送方会重新发送。

此外，TCP 还有滑动窗口和流量控制，避免发送方把接收方缓冲区打满；拥塞控制避免发送方把整个网络打爆。最大报文段长度 MSS 也会控制每次发送的数据大小。

---

### 38. 说说 TCP 的流量控制？

TCP 流量控制是为了避免发送方发送太快，把接收方的缓冲区撑爆。

它主要通过滑动窗口实现。接收方会在 ACK 报文中带上自己的接收窗口大小，也就是告诉发送方：“我现在还能接收多少数据”。发送方根据这个窗口大小控制发送速度。

如果接收方处理数据比较慢，缓冲区剩余空间变小，就会把窗口调小；如果缓冲区满了，窗口可能变成 0，发送方就会暂停发送。等接收方处理完一些数据，再通知发送方窗口变大，发送方继续发送。

所以流量控制关注的是接收方能力，防止发送方压垮接收方。

---

### 39. 详细说说 TCP 的滑动窗口？

如果 TCP 每发一个包都要等 ACK 才发下一个，效率会很低。滑动窗口就是为了解决这个问题。

发送方可以在窗口范围内连续发送多个数据段，不需要每个都等确认。随着 ACK 返回，窗口向前滑动，发送方就可以继续发送后面的数据。

发送窗口一般可以分成几部分：已经发送并确认的数据、已经发送但未确认的数据、可以发送但还没发送的数据、暂时不能发送的数据。

接收窗口表示接收方缓冲区还能接收多少数据。TCP 报文头中的 window 字段就是用来通知对方窗口大小的。滑动窗口既提高了传输效率，也实现了流量控制。

---

### 40. 了解 Nagle 算法和延迟确认吗？

Nagle 算法和延迟确认都是为了解决小包太多导致网络效率低的问题。

Nagle 算法的核心是：在任意时刻，最多只能有一个未被确认的小报文段。如果已经有一个小包发出但还没收到 ACK，后续小数据会先攒起来，等收到 ACK 或攒到一定大小再发送。

延迟确认是接收方收到数据后，不一定马上发 ACK，而是稍微等一会儿，看能不能和要发送的数据一起带回去，或者合并多个 ACK，减少报文数量。

但是这两个机制叠加时，可能会导致延迟变大。所以一些低延迟场景，比如实时交互，会关闭 Nagle 算法，常见设置是 `TCP_NODELAY`。

---

### 41. 说说 TCP 的拥塞控制？

拥塞控制是为了避免发送方把整个网络打爆。它和流量控制不同，流量控制看的是接收方缓冲区，拥塞控制看的是整个网络的拥堵情况。

TCP 拥塞控制主要有四个阶段：慢启动、拥塞避免、快重传和快恢复。

慢启动阶段，拥塞窗口 cwnd 从小开始指数增长，快速探测网络承载能力。达到慢启动门限 ssthresh 后进入拥塞避免阶段，窗口线性增长，变得更保守。

如果发生超时重传，TCP 会认为网络拥塞比较严重，把门限降低，拥塞窗口重新变小。如果收到三个重复 ACK，会触发快重传，不等超时就重发丢失的数据，并进入快恢复阶段。

---

### 42. 说说 TCP 的重传机制？

TCP 常见重传机制有超时重传、快速重传、SACK 和 D-SACK。

超时重传是发送数据后启动计时器，如果超过 RTO 还没收到 ACK，就认为数据丢了并重传。RTO 不能太大，也不能太小，一般会根据 RTT 动态计算。

快速重传是接收方连续收到后面的数据，但中间缺了一段，就会反复发送相同 ACK。发送方收到三个重复 ACK，就认为某个报文丢了，马上重传，不用等超时。

SACK 是选择性确认，接收方告诉发送方哪些数据收到了，哪些没收到，让发送方只重传缺失部分。D-SACK 可以告诉发送方哪些数据被重复接收，用来判断是否发生了误重传。

---

### 43. 说说 TCP 的粘包和拆包？

TCP 是面向字节流的协议，它不关心应用层所谓的“一个完整消息”边界。所以从业务角度看，就可能出现粘包和拆包。

粘包是多个小消息被 TCP 合在一起发出去，接收方一次读到多个业务包。拆包是一个完整业务包被拆成多个 TCP 段发送，接收方需要多次读取才能拿完整。

产生原因包括：应用写入数据太小，TCP 缓冲区合并发送；接收方没有及时读取；数据超过 MSS 被拆分；或者发送缓冲区剩余空间不足。

解决方法一般在应用层做协议设计，比如固定消息长度、使用分隔符、在消息头里加长度字段，或者使用 TLV 格式。

---

### 44. 说说 TCP 和 UDP 的区别？

最核心的区别是：TCP 面向连接，UDP 无连接。

TCP 在传输前需要三次握手，传输过程中有确认、重传、排序、流量控制和拥塞控制，所以可靠但开销更大。UDP 不需要建立连接，也不保证可靠性、顺序和不丢包，但是头部开销小、速度快、延迟低。

应用场景上，TCP 适合文件传输、网页访问、邮件、远程登录等要求可靠性的场景。UDP 适合直播、语音、视频、游戏、DNS 等更看重实时性和低延迟的场景。

---

### 45. 为什么 QQ 采用 UDP 协议？

准确说，QQ 并不是完全使用 UDP。比如文件传输这类要求可靠性的功能，可能会用 TCP。语音、视频、消息通知这类场景更适合 UDP。

UDP 的优势是延迟低、开销小，不需要三次握手，也没有复杂的重传和拥塞控制。对语音视频来说，偶尔丢一帧或者声音卡一下可以接受，但如果为了重传导致几秒后才播放旧数据，体验反而更差。

另外，QQ 这类系统用户量很大，UDP 在服务端连接维护上的成本更低，更适合大规模并发场景。当然业务层也可以自己做一些确认、重发和补偿机制。

---

### 46. UDP 协议为什么不可靠？

UDP 不可靠是因为它只负责把数据报尽力发出去，但不保证对方一定收到。

具体来说，UDP 传输前不建立连接，发送后不需要确认，不维护连接状态，也没有超时重传、流量控制、拥塞控制和包顺序保证。报文可能丢失、重复、乱序，UDP 本身都不处理。

如果业务需要可靠性，就要在应用层自己实现确认、重传、序号、去重和超时控制。

---

### 47. DNS 为什么要用 UDP？

更准确地说，DNS 既使用 UDP，也使用 TCP。

普通域名查询一般使用 UDP，因为一次 DNS 查询和响应的数据量通常比较小，UDP 不需要建立连接，速度快、开销低，很适合 DNS 这种高频、短小的请求。

但在区域传送，或者响应数据很大超过 UDP 限制时，DNS 会使用 TCP。因为 TCP 能保证可靠传输，也能传输更大的数据。

普通 DNS 查询主要用 UDP，区域传送和大报文场景会用 TCP。

---

## IP

### 48. IP 协议的定义和作用？

IP 协议是网络层的核心协议，负责在不同网络之间传输数据包。它提供的是无连接、不可靠、尽力而为的传输服务。

IP 的主要作用有两个：寻址和路由。寻址是通过源 IP 和目的 IP 标识数据从哪里来、到哪里去；路由是中间路由器根据目的 IP 和路由表选择下一跳，把数据包一步步转发到目标网络。

IP 本身不保证可靠性，它只尽力把包送到。可靠传输通常由上层 TCP 来负责。

---

### 49. IP 地址有哪些分类？

传统 IPv4 地址可以分成 A、B、C、D、E 五类。

A 类地址网络号占 8 位，主机号占 24 位，适合超大型网络。B 类地址网络号占 16 位，主机号占 16 位，适合中型网络。C 类地址网络号占 24 位，主机号占 8 位，适合小型网络。

D 类地址用于多播，E 类地址保留实验使用。

不过实际工程中，现在更常用 CIDR，也就是无类别域间路由，比如 `192.168.1.0/24`，不再严格按照 A、B、C 类来划分。

---

### 50. 域名和 IP 的关系？一个 IP 可以对应多个域名吗？

IP 地址用来定位网络上的一台主机，域名是为了方便人记忆而起的名字。DNS 负责把域名解析成 IP。

一个域名可以对应多个 IP，这通常用于负载均衡，比如同一个域名解析到多台服务器。客户端实际访问时，DNS 可能返回其中一个 IP。

一个 IP 也可以对应多个域名。比如一台服务器上部署多个网站，多个域名都解析到同一个 IP，再通过 Host 头区分访问哪个网站。

---

### 51. IPv4 地址不够如何解决？

IPv4 地址只有 32 位，全球设备越来越多，所以地址不够用。常见解决方式有 DHCP、CIDR、NAT 和 IPv6。

DHCP 可以动态分配 IP，让设备接入网络时再拿地址，提高地址利用率。CIDR 取消传统 A、B、C 类地址划分，让 IP 地址分配更灵活。

NAT 是非常常见的方式，局域网内部使用私有 IP，访问公网时由网关转换成公网 IP。这样大量内网设备可以共享少量公网 IP。

根本解决方案是 IPv6，它使用 128 位地址，地址空间远远大于 IPv4。

---

### 52. 说下 ARP 协议的工作过程？

ARP 是地址解析协议，用来根据 IP 地址查找对应的 MAC 地址。

每台主机都会维护一个 ARP 缓存表，记录 IP 到 MAC 的映射。当源主机要给目标 IP 发数据时，先查 ARP 缓存。如果有对应 MAC，就直接封装数据帧发送。

如果缓存里没有，源主机会在局域网内广播 ARP 请求，意思是“这个 IP 是谁？请告诉我你的 MAC 地址”。目标主机收到后，发现请求的 IP 是自己，就单播返回 ARP 响应，告诉源主机自己的 MAC 地址。

源主机收到响应后，把这个 IP 和 MAC 映射存入 ARP 缓存，后续通信就可以直接使用。

---

### 53. 为什么既有 IP 地址，又有 MAC 地址？

IP 地址和 MAC 地址属于不同层，解决的问题也不同。

MAC 地址是数据链路层使用的物理地址，主要用于局域网内相邻设备之间的数据帧传输。IP 地址是网络层使用的逻辑地址，用于跨网络寻址和路由。

如果只有 MAC 地址，路由器就要知道全世界所有设备的 MAC 地址，这几乎不可能。IP 地址具有层次结构，可以根据网络号进行路由聚合，方便在互联网中转发。

所以可以理解为：IP 负责把数据送到目标网络，MAC 负责在当前链路上把数据送到下一跳设备。

---

### 54. ICMP 协议的功能？

ICMP 是网际控制报文协议，属于网络层，主要用于传递错误报告和控制信息。

比如当目标主机不可达、路由不可达、TTL 超时、需要分片但不能分片等情况发生时，网络设备可以通过 ICMP 把错误信息反馈给源主机。

我们常用的 ping 就是基于 ICMP。ping 会发送 ICMP Echo Request，如果目标主机可达，就会返回 ICMP Echo Reply。

所以 ICMP 不是用来传输业务数据的，而是用来辅助 IP 网络进行诊断和错误反馈。

---

### 55. 说下 ping 的原理？

ping 用来测试目标主机是否可达，底层依赖 ICMP 协议。

当机器 A ping 机器 B 时，A 会构造一个 ICMP Echo Request 报文，把它交给 IP 层。IP 层加上源 IP 和目的 IP 后发送出去。如果目标 IP 不在同一局域网，还需要经过路由器转发。

在局域网内真正发送前，还需要通过 ARP 获取下一跳或目标主机的 MAC 地址。目标主机收到 ICMP 请求后，如果允许响应，就返回 ICMP Echo Reply。

ping 根据是否收到响应、响应时间、丢包率等信息判断网络连通性和延迟情况。

## 网络安全

### 56. 说说有哪些安全攻击？

网络攻击大致可以分为被动攻击和主动攻击。

被动攻击是攻击者只窃听或分析通信内容，不直接修改数据。比如窃听用户账号密码、分析流量模式等。这类攻击因为不改变数据，所以比较难被发现。

主动攻击会直接影响数据或服务。常见形式包括篡改报文、伪造身份、恶意程序、病毒、木马、蠕虫、拒绝服务攻击等。

从防护角度看，可以通过加密、防火墙、认证、权限控制、入侵检测、限流和日志审计等手段降低风险。

---

### 57. DNS 劫持了解吗？

DNS 劫持就是攻击者篡改域名解析结果，让用户访问域名时被解析到错误的 IP。

比如用户访问正常网站，但 DNS 返回了攻击者控制的服务器 IP，用户就可能进入钓鱼网站。也可能直接导致用户无法访问目标网站。

DNS 劫持可能发生在本机 hosts 文件、本地路由器、运营商 DNS 或域名注册信息被篡改等环节。

防护方式包括使用可信 DNS、HTTPS、DNSSEC、检查域名证书、避免使用不可信网络，以及监控域名解析结果是否异常。

---

### 58. 什么是 CSRF 攻击？如何避免？

CSRF 是跨站请求伪造。它利用的是用户已经登录某网站，浏览器会自动携带 Cookie 这一点，诱导用户在不知情的情况下向目标网站发送恶意请求。

比如用户已经登录银行网站，然后访问了攻击者页面，攻击者页面里隐藏了一个转账请求。浏览器发送请求时自动带上银行的登录 Cookie，银行服务器可能误以为这是用户本人操作。

防护方式包括：使用 CSRF Token，让请求必须带上攻击者拿不到的随机 token；校验 Referer 或 Origin；关键操作要求二次确认；Cookie 设置 SameSite；重要接口不要只依赖 Cookie 进行身份判断。

---

### 59. 什么是 DoS、DDoS、DRDoS 攻击？

DoS 是拒绝服务攻击，攻击者通过大量请求或异常请求让目标服务不可用。比如耗尽服务器 CPU、内存、连接数或带宽。

DDoS 是分布式拒绝服务攻击，攻击者控制大量机器同时攻击目标，规模更大，更难防御。常见类型有 SYN Flood、UDP Flood、ACK Flood、Ping of Death 等。

DRDoS 是分布式反射拒绝服务攻击。攻击者伪造受害者 IP，向第三方服务器发送请求，让这些服务器把大量响应流量反射到受害者机器 

防护方式包括限流、黑白名单、流量清洗、CDN、高防 IP、SYN Cookie、连接数限制和监控告警。

---

### 60. 什么是 XSS 攻击，如何避免？

XSS 是跨站脚本攻击，攻击者把恶意 JavaScript 注入到网页里，当用户浏览页面时，恶意脚本在用户浏览器中执行。

常见类型有存储型 XSS、反射型 XSS 和 DOM 型 XSS。存储型是恶意脚本被存进服务器，比如评论区；反射型是恶意脚本通过 URL 参数反射到页面；DOM 型是前端脚本处理不当导致恶意代码执行。

防护方式主要有：对用户输入做过滤和校验，对输出做 HTML 转义；避免直接拼接 HTML；设置 HttpOnly Cookie 防止脚本读取 Cookie；使用 CSP 内容安全策略；前端框架中避免危险 API，比如 `innerHTML`。

---

### 61. 对称加密与非对称加密有什么区别？

对称加密使用同一个密钥进行加密和解密。它的优点是速度快，适合大量数据加密；缺点是密钥分发困难，因为通信双方要先安全地拿到同一个密钥。常见算法有 AES、DES。

非对称加密使用一对密钥，公钥和私钥。公钥可以公开，私钥自己保存。用公钥加密的数据只能用私钥解密。它的优点是密钥分发更安全，缺点是计算速度慢。常见算法有 RSA。

实际系统里通常会混合使用。比如 HTTPS 握手阶段用非对称加密协商对称密钥，后续数据传输用对称加密提高效率。

---

### 62. RSA 和 AES 算法有什么区别？

RSA 是非对称加密算法，使用公钥加密、私钥解密，或者私钥签名、公钥验签。它的安全性依赖大数分解难题，但计算成本比较高，不适合大量数据直接加密。

AES 是对称加密算法，加密和解密使用同一个密钥。它速度快，适合加密大量数据，也更容易用硬件加速。

所以在实际通信中，经常用 RSA 或其他非对称算法解决密钥交换和身份认证问题，再用 AES 加密真正传输的数据。简单说，RSA 解决“怎么安全交换密钥”，AES 解决“怎么高效加密数据”。

# 最后总复习

这份计算机网络面试题可以按这条主线记：

1. 网络分层：OSI、TCP/IP、五层模型。
2. HTTP：状态码、请求方法、GET/POST、报文结构、版本演进、HTTPS。
3. TCP：三次握手、四次挥手、可靠性、滑动窗口、流量控制、拥塞控制、重传、粘包拆包。
4. UDP：无连接、不可靠、低延迟，适合实时场景。
5. IP：寻址、路由、ARP、ICMP、ping。
6. 安全：HTTPS、DNS 劫持、CSRF、XSS、DDoS、加密算法。

面试回答时不要只背概念，最好用“它解决什么问题、怎么解决、有什么缺点、项目里怎么用”这四步展开。
