# 分布式面试口语版

## 1. 说说 CAP 原则？

CAP 原则指的是分布式系统里的三个核心指标：一致性、可用性、分区容错性。
 一致性就是多个节点上的数据要保持一致；可用性就是每次请求都能得到正常响应；分区容错性就是即使网络出现分区，系统也要继续对外提供服务。

但是 CAP 不能三者同时满足，最多只能同时满足两个。在分布式系统里，网络分区是无法完全避免的，所以 P 基本是必须保证的。这样一来，就只能在 C 和 A 之间做取舍：要么保证一致性，牺牲一部分可用性；要么保证可用性，接受短时间的数据不一致。

## 2. 为什么 CAP 不可兼得？

因为在分布式系统里，网络分区是很常见的，比如两个节点之间网络不通了。假设用户先访问节点 N1 修改了数据，然后下一次请求落到了节点 N2，而 N2 还没同步到最新数据。

这个时候如果要保证一致性，就不能让 N2 返回旧数据，那请求可能就要阻塞或者失败，可用性就受影响。
 如果要保证可用性，N2 立即返回结果，但返回的可能是旧数据，一致性就受影响。

所以在发生网络分区时，一致性和可用性天然会冲突，这就是 CAP 不可兼得的原因。

## 3. CAP 对应的模型和应用？

CAP 常见模型有 CA、CP、AP。

CA 是放弃分区容错，保证一致性和可用性，但严格来说在真正分布式环境里，网络分区不可避免，所以纯 CA 很难存在。

CP 是保证一致性和分区容错，牺牲一部分可用性。比如 ZooKeeper 更偏 CP，分布式锁、强一致场景一般也更偏 CP。

AP 是保证可用性和分区容错，牺牲强一致性，接受最终一致。比如 Eureka 更偏 AP，很多缓存系统、DNS、部分 NoSQL 系统也偏 AP。

所以面试里可以总结一句：**分布式系统一般 P 是必须的，剩下就是根据业务在 C 和 A 之间取舍。**

## 4. BASE 理论了解吗？

BASE 理论可以看成是对 CAP 中 AP 模型的一种工程化落地。它的核心思想是：既然很多业务没办法做到强一致，那就退一步，保证最终一致。

BASE 包括三个部分：基本可用、软状态、最终一致性。
 基本可用是说系统出故障时，核心功能还能用，只是可能响应慢一点，或者部分功能降级。
 软状态是说系统中间可以存在短暂的不一致状态。
 最终一致性是说经过一段时间后，多个节点的数据最终要达到一致。

比如电商下单后，订单、库存、积分这些系统不一定每一刻都强一致，但通过消息队列、重试、补偿机制，最终能达到一致，这就是 BASE 的思想。

## 5. 有哪些分布式锁的实现方案？

常见的分布式锁有三种实现方式：MySQL、ZooKeeper 和 Redis。

MySQL 实现比较简单，一般是建一张锁表，对锁名称加唯一索引。加锁就是插入一条记录，释放锁就是删除记录。优点是实现简单，缺点是性能一般，高并发下数据库压力比较大。

ZooKeeper 可以通过临时顺序节点实现锁。多个客户端都在同一个目录下创建节点，谁的节点序号最小，谁就拿到锁。释放锁就是删除自己的节点。它的可靠性比较好，但组件偏重。

Redis 是现在用得最多的方式，一般用 `set key value nx ex seconds` 实现。`nx` 保证只有 key 不存在时才能加锁，`ex` 设置过期时间防止死锁。生产里通常用 Redisson，因为它封装了可重入锁、自动续期这些能力。

## 6. 什么是分布式事务？

本地事务一般是一个数据库内部的事务，可以依靠数据库本身保证 ACID。
 但是分布式事务涉及多个服务、多个数据库，比如下单要改订单库、扣库存、调用支付服务，这些操作不在同一个数据库里，单靠本地事务就不够了。

所以分布式事务要解决的核心问题就是：多个节点上的操作，要么最终都成功，要么最终都失败，避免出现订单创建成功了，但是库存没扣，或者支付成功了但订单没更新这种数据不一致问题。

## 7. 分布式事务有哪些常见实现方案？

常见方案有 2PC、3PC、TCC、本地消息表、MQ 事务消息、最大努力通知、Saga，还有像 Seata 这种框架。

### 7.1 说说 2PC 两阶段提交？

2PC 就是两阶段提交，主要分为准备阶段和提交阶段。
 第一阶段，事务协调者询问所有参与者：你们能不能提交？参与者执行预提交，并返回 yes 或 no。
 第二阶段，如果所有参与者都返回 yes，协调者就通知大家提交；只要有一个失败，就通知大家回滚。

它的优点是比较容易理解，能尽量保证强一致性。缺点也明显：第一，协调者是单点，如果协调者挂了，参与者可能一直阻塞；第二，资源在事务过程中会被锁住，性能不好；第三，在第二阶段如果部分节点收到 commit，部分节点没收到，也可能出现数据不一致。

### 7.2 3PC 了解吗？

3PC 是对 2PC 的改进，把流程拆成了 CanCommit、PreCommit、DoCommit 三个阶段。

CanCommit 是先询问能不能提交；PreCommit 是预提交；DoCommit 是真正提交。
 相比 2PC，3PC 加入了超时机制，主要是为了减少 2PC 的同步阻塞问题。

但是 3PC 也不能百分百保证分布式系统数据一致。它只是比 2PC 更进一步缓解了阻塞和单点问题，实际项目里用得也没有 TCC、MQ 最终一致、Seata 这些方案多。

### 7.3 TCC 了解吗？

TCC 是 Try、Confirm、Cancel。它可以理解成业务层面的两阶段提交。

Try 阶段先做资源预留，比如订单状态改成支付中，库存先冻结。
 Confirm 阶段是在 Try 成功后真正确认，比如订单改成支付成功，冻结库存真正扣减。
 Cancel 阶段是在失败时做补偿，比如订单改成支付失败，把冻结库存释放回去。

TCC 的优点是不会长期锁数据库资源，性能比数据库层面的 2PC 好，也更适合高并发业务。缺点是业务侵入性比较强，因为每个接口都要自己实现 Try、Confirm、Cancel，开发成本比较高。

### 7.4 本地消息表了解吗？

本地消息表的核心思想是把分布式事务拆成本地事务加消息补偿。

比如订单服务创建订单时，同时在本地消息表里插入一条消息，这两个操作放在同一个本地事务里提交。然后定时任务扫描消息表，把消息发送到 MQ。库存服务消费 MQ 消息后扣减库存，处理成功后再更新消息状态。

这种方式能保证最终一致性，但缺点是业务系统里要额外维护消息表，还要有定时扫描、重试、告警机制，整体实现会稍微重一些。

### 7.5 MQ 消息事务了解吗？

MQ 事务消息的核心是保证“本地事务”和“消息发送”一致。

一般流程是：先发送一条半消息到 MQ；然后执行本地事务；如果本地事务成功，就提交消息，MQ 再把消息投递给消费者；如果本地事务失败，就回滚消息，MQ 删除这条半消息。

比如 RocketMQ 就支持事务消息。它相比本地消息表，少了自己建消息表和定时扫描的工作，对业务侵入更小。但它依赖消息中间件本身支持事务消息。

### 7.6 最大努力通知了解吗？

最大努力通知适合对一致性实时性要求不高的场景，比如支付结果通知、短信通知。

比如业务系统调用支付平台支付，支付平台处理完成后会尽量通知业务系统。如果通知失败，就按照规则重试，比如重试几次、间隔多久。达到最大重试次数后还失败，就不再主动通知。业务系统可以通过查询接口，主动去支付平台查最终结果。

所以它不是强一致，而是通过“通知 + 重试 + 查询补偿”来保证最终一致。

## 8. 你们用什么？能说一下 Seata 吗？

可以这样回答：我们项目里如果要处理分布式事务，比较常见的是用 Seata，因为自己实现事务协调、回滚补偿会比较复杂。

Seata 里面有三个核心角色：TC、TM、RM。
 TC 是事务协调者，负责管理全局事务状态，决定提交还是回滚。
 TM 是事务管理器，负责开启、提交或回滚全局事务。
 RM 是资源管理器，负责本地分支事务，并向 TC 上报状态。

整体流程是：服务 A 先通过 TM 向 TC 开启全局事务，TC 返回一个 XID。服务 A 执行本地分支事务，并把分支事务注册到 TC。然后服务 A 调用服务 B，XID 会在调用链中传递，服务 B 也注册自己的分支事务。最后业务执行完后，TM 通知 TC 提交或回滚，TC 再协调所有分支事务一起提交或回滚。

## 9. 分布式算法 Paxos 了解吗？

Paxos 是一种分布式一致性算法，用来解决多个节点之间如何对某个值达成一致的问题。它有三个角色：Proposer 提议者、Acceptor 接受者、Learner 学习者。

Paxos 的流程主要分为 Prepare 和 Accept 两个阶段。
 Prepare 阶段，提议者先发起一个提案编号，询问多数 Acceptor 能不能接受。Acceptor 如果发现这个编号比之前见过的都大，就承诺不再接受更小编号的提案。
 Accept 阶段，如果提议者收到了超过半数的响应，就带着最终 value 发起 Accept 请求。只要多数 Acceptor 接受，这个提案就算达成共识。

Paxos 的优点是理论非常严谨，容错能力强。缺点是比较难理解，实现复杂。如果多个提议者同时竞争，Basic Paxos 可能会反复冲突，所以后来有 Multi Paxos，通过选 Leader 来减少冲突。

## 10. 说说 Raft 算法？

Raft 也是一种一致性算法，目标和 Paxos 类似，但 Raft 最大特点是更容易理解。

Raft 里面有三个角色：Leader、Follower、Candidate。刚开始所有节点都是 Follower，如果一段时间没有收到 Leader 心跳，就会变成 Candidate 发起选举。Candidate 会给自己投票，并向其他节点请求投票。如果拿到超过半数的票，就成为 Leader。

Leader 选出来之后，所有写请求都由 Leader 处理，然后同步日志给 Follower。Leader 会定期发送心跳，告诉大家自己还活着。如果 Follower 长时间收不到心跳，就会认为 Leader 挂了，然后重新发起选举。

所以 Raft 可以简单总结成三件事：Leader 选举、日志复制、安全提交。

## 11. 说说什么是幂等性？

幂等性就是同一个接口，多次调用和调用一次的结果是一样的。简单说，就是**多次调用如一次**。

比如用户提交订单时，因为网络超时，前端或调用方重试了几次。如果接口没有做幂等，就可能生成多条订单。再比如 MQ 消费消息时，消息可能重复投递，如果消费者没有做幂等，就可能重复扣库存。

所以在分布式系统里，只要接口涉及写操作，比如新增、扣减、支付、发券，都要考虑幂等性。

### 怎么保证接口幂等性？

常见方式有几种。

第一，insert 前先 select，根据 requestId 或业务唯一号查一下，如果已经处理过，就直接返回。
 第二，加唯一索引，比如订单号、流水号加唯一索引，重复插入会失败。
 第三，加悲观锁，比如更新账户余额时 `select for update` 锁住这一行。
 第四，加乐观锁，比如加 version 字段，更新时带上版本号，更新成功后版本号加一，重复请求就更新不了。
 第五，建防重表，比如 MQ 消费时，把 messageId 存到防重表，消费前先查是否处理过。
 第六，用状态机，比如订单只能从“待支付”流转到“已支付”，不能重复支付。
 第七，用分布式锁，比如 Redis 或 Redisson 控制同一时间只有一个请求处理。
 第八，用 token 机制，请求前先获取唯一 token，提交时校验 token 是否存在，用完就删除。 

## 12. 你了解哪些限流算法？

常见限流算法有计数器、漏桶算法和令牌桶算法。

计数器最简单，比如限制 1 秒最多 100 个请求。每来一个请求计数加一，超过阈值就拒绝。缺点是可能有临界突刺，比如前 10ms 就打满了 100 个请求，后面 990ms 都只能拒绝。

漏桶算法可以理解成水从桶底匀速流出，请求进来的速度可以不均匀，但处理速度是固定的。请求太多时先放到桶里，也就是队列里，桶满了才丢弃。它的优点是流量比较平滑。

令牌桶算法是系统按固定速率往桶里放令牌，请求来了必须先拿到令牌才能通过。没有令牌就被限流。令牌桶相比漏桶，更能支持一定程度的突发流量。比如 Guava 的 RateLimiter 就是基于令牌桶思想实现的。
